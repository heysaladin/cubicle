# Cubicle DS — Architecture

This document explains how the pieces of Cubicle fit together so you (and anyone joining the team) never have to guess "where does this belong" again.

## The mental model

There is **one source of truth for tokens** (colors, type, spacing, radius, shadows): `tokens/*.json`. Everything else — Figma styles, CSS variables, Swift constants, Kotlin theme, Dart tokens — is **generated from that JSON**. No hand-maintained copies anywhere.

Components are the next layer up. They consume tokens and live **per platform**, hand-written, but their styling never hard-codes hex values or pixel numbers. Always `var(--color-primary)`, never `#171717`.

## Folder responsibilities

### `tokens/` (this folder)
The source of truth. Hand-edited JSON in [DTCG format](https://design-tokens.github.io/community-group/format/). Compatible with both Style Dictionary v4+ and the Tokens Studio Figma plugin.

Split into:
- `core.json` — raw palette, font sizes, spacing, radii, shadows. Platform-agnostic primitives.
- `semantic.light.json` — semantic aliases for light mode (`primary`, `background`, `border`, etc.) that *reference* core tokens.
- `semantic.dark.json` — same aliases for dark mode.

Edit either in Figma (via Tokens Studio's GitHub sync) or directly here. Run `npm run tokens:build` to regenerate platform outputs.

### `cubicle-storybook/` (your existing repo)
The **canonical DS codebase**. Holds:
- All shadcn React components in `src/components/ui/`
- Storybook stories — your living documentation
- A generated `src/styles/tokens.css` (regenerated by Style Dictionary, never hand-edited)
- The published npm package (when you're ready)

### `cubicle-claudedesign/`
Becomes a **thin AI skill only** — keep `SKILL.md`, `colors_and_type.css`, `assets/`, fonts, and the static HTML previews so Claude can generate Cubicle-branded mocks/throwaways.

**Delete from this folder:**
- `ui_kits/website/`
- `ui_kits/mobile/`
- `ui_kits/flutter/`
- `ui_kits/ios/`
- `ui_kits/android/`
- `ui_kits/cubicle/`

These were hand-maintained ports drifting out of sync. Their replacements are *generated* from `tokens/*.json` via Style Dictionary, in their proper home (each platform's real codebase).

### Figma file
The visual playground. Connected to `tokens/*.json` via the **Tokens Studio** plugin (free, open-source). Designers can:
- Edit tokens in Figma → push to `tokens/` repo → CI rebuilds platforms
- Pull token changes from `tokens/` repo → Figma styles update automatically

This is what kills the Figma↔code drift permanently.

## The flow when a token changes

**Designer changes "primary" from neutral-900 to neutral-800 in Figma:**
1. Designer pushes via Tokens Studio → opens PR on the tokens repo
2. PR reviewed, merged
3. CI runs `npm run tokens:build` → regenerates all platform outputs
4. Each platform's CI picks up the new files; web, iOS, Android, Flutter all reflect the change
5. Storybook visual diff (Chromatic) flags any unexpected component impact

**Engineer changes spacing-md from 16px to 14px in `tokens/core.json`:**
1. Open PR with the JSON change
2. Style Dictionary regenerates everything
3. Tokens Studio in Figma pulls the change → Figma styles update
4. Same Chromatic visual review

No more manual sync. No more "the Figma says 16 but the CSS says 14."

## The flow when a component changes

Components live in `cubicle-storybook/src/components/ui/`. They are **never** ported by hand to other platforms — each platform writes its own component using its own framework, but all read from the same generated tokens.

To add or modify a component:
1. Update the React component in `cubicle-storybook`
2. Update or add a story in `src/stories/`
3. Document any new prop variants in JSDoc above the component
4. Open PR — Chromatic captures the visual diff
5. If the component needs a Figma counterpart, designer creates it in Figma using the same tokens (so it'll naturally match)

iOS/Android/Flutter teams build their own equivalent component when they need it, against the generated token files for their platform. No code is shared across platforms — only tokens.

## Why this works

- **One brain to update.** Change a token once, propagates everywhere.
- **No copy-paste drift.** Style Dictionary is deterministic.
- **Designers stay in Figma.** They never touch JSON unless they want to.
- **Engineers stay in code.** They never have to reverse-engineer a Figma file.
- **Components stay native.** No "lowest common denominator" cross-platform UI library.

## What this does NOT solve

- Component parity across platforms — that's a team discipline issue, not a tooling one. A weekly DS sync and a shared Notion/Linear board for component status helps.
- Animation / motion tokens — DTCG support is still maturing. For now keep motion in code.
- Brand assets (logos, illustrations) — those live in `cubicle-claudedesign/assets/` and are referenced directly, not tokenized.
